# End-to-End MLOps Homework: Spotify Genre Classification & Drift

Welcome to the final assignment for the MLOps Fundamentals and Practice course!

In this homework, you will implement a complete, production-ready MLOps pipeline for a tabular dataset (Spotify Audio Features). You will manage data versioning, orchestrate experiments, deploy an API, and monitor for data drift.

## Project Structure (Monorepo)
We are using a monorepo approach, separating the machine learning pipeline, the serving infrastructure, and the monitoring scripts into distinct Python projects.

```text
Homework_E2E/
├── .github/workflows/   # CI/CD pipelines
├── data_pipeline/       # DVC orchestrated ML training pipeline
│   ├── src/             # Scripts: download, process, train, evaluate
│   ├── tests/           # Unit tests for the pipeline steps
│   ├── dvc.yaml         # Pipeline definition
│   └── params.yaml      # Hyperparameters for MLflow experiments
├── model_serving/       # FastAPI application and Docker deployment
│   ├── app/             # FastAPI code
│   ├── tests/           # API integration tests
│   └── Dockerfile       # Container definition (bakes in the best model)
└── drift_monitoring/    # Scripts for offline batch drift detection
    └── src/
```

## Your Tasks

### 1. Data Pipeline & Orchestration (DVC + MLflow)
Located in `data_pipeline/`.
*   **Download & Process:** Complete the scripts to download the dataset (ensure the path is parameterized) and split it temporally (pre-2018 for training/validation, 2018+ for simulating production).
*   **Experimentation:** Complete `train.py` to log to MLflow. You must train at least two types of models (e.g., Logistic Regression and XGBoost). Use `dvc.yaml` and `params.yaml` to orchestrate hyperparameter tuning.
*   **Evaluation:** Complete `evaluate.py` to programmatically compare the MLflow runs, select the best model based on accuracy, and assign it a specific **alias** (e.g., `@champion`) in the MLflow Model Registry.

### 2. Model Serving (FastAPI + Docker)
Located in `model_serving/`.
*   **API Implementation:** Build a FastAPI `/predict` endpoint that accepts song features as a JSON payload.
*   **Model Baking:** The `Dockerfile` must include a build step that pulls the model with the `@champion` alias from MLflow so that the model artifact is baked into the immutable container image.
*   **Data Logging:** The API must log all incoming inference requests (the features) to a local file (e.g., `logs/api_requests.jsonl`). This simulates capturing production data for drift analysis.

### 3. Drift Analysis (Batch)
Located in `drift_monitoring/`.
*   **Offline Drift Detection:** Write a script that compares the statistical distribution of the training dataset (tracked by DVC) against the incoming logs generated by your FastAPI service. You will use the 2018+ data split to simulate traffic hitting your API.

### 4. CI/CD (GitHub Actions)
*   Ensure the provided unit tests in both `data_pipeline/tests` and `model_serving/tests` pass.
*   Configure `.github/workflows/ci.yml` to automatically run `flake8` and `pytest` on every Pull Request to `main`.
